# micron Code Review & Recommendations

> Generated by moonshotai/kimi-k2.7-code via OpenRouter  
> Date: 2026-07-07

## Executive Summary

`micron` is a well-structured, lightweight agent framework. The architecture (file-based memory, Markdown skills, pluggable LLM backends) is sound and aligned with the stated goal of running on 1B models on CPU. However, several areas need attention before it can be considered production-ready or robust for diverse local/API models:

1. **Tool-calling protocol mismatch**: The system tries to support both native `tools` (OpenAI format) and text-based parsing, but the prompt hard-codes text markup instructions that confuse API-level tool callers.
2. **Message history is incorrect**: Tool results are injected as raw `user` messages instead of the OpenAI-required `assistant`/`tool` message pattern.
3. **Tool return values are strings**: Fine for humans, but breaks programmatic consumers and makes multi-tool reasoning harder.
4. **Security holes**: Path-traversal guard uses simple string prefix checks; `python_eval` blocklists are bypassable.
5. **Memory TF-IDF bug**: Operator precedence in IDF calculation likely produces wrong scores.
6. **No tests**: Critical for a framework that loads external Markdown skills and executes shell commands.

The code is roughly **300–400 lines of clean core plus ~500 lines of tools** — very hackable, but needs a couple of focused refactoring passes.

## Detailed Findings

### 1. `micron/agent.py` — Core Agent Loop

#### What works
- Clear separation of concerns (LLM, memory, skills, tools, prompt builder).
- Loop detection with fingerprinting.
- Pivot-thought injection after 3 consecutive failures.
- History compression for long conversations.
- Text-based tool-call fallback for models without native tool support.

#### Issues

**A. Message history is not OpenAI-correct**

Current code appends an `assistant` message with `content=full_text`, then a `user` message saying "Tool results:...". The OpenAI/llama.cpp tool protocol expects:

```json
{"role": "assistant", "content": null, "tool_calls": [{"id": "call_1", "type": "function", "function": {"name": "x", "arguments": "{}"}}]}
{"role": "tool", "tool_call_id": "call_1", "content": "<result>"}
```

**Impact**: API models (GPT-4, Claude, OpenRouter endpoints) receive garbled context; local models with `tools` support get ambiguous signal.  
**Fix**: Track `tool_call_id`s and emit proper `tool` role messages. Keep assistant message empty/null when only tools were emitted.

---

**B. `_continue_conversation()` recursively calls `run()`**

```python
def _continue_conversation(self, user_message: str, history: list[dict] | None = None):
    yield from self.run(user_message, history=history)
```

This re-builds the system prompt, re-injects memory/knowledge, and loses `tools_used_this_turn` state. It also risks duplicate prompts after a write-tool confirmation.

**Fix**: Maintain a single conversation state object / queue, or refactor `run()` into smaller phases: `build_messages`, `stream`, `execute`, `loop`.

---

**C. Text fallback interacts badly with native tool callers**

```python
if not pending_calls and full_text and not tools_used_this_turn:
    text_calls = self._parse_text_tool_calls(full_text)
```

If an API model returns text that happens to contain `name="read_file">`, it gets parsed even though the model did not intend a tool call. Also, `tools_used_this_turn` only tracks read calls — if a write call needs confirmation, the flag is never set.

**Fix**: Gate text parsing by provider capability. Add a config flag `use_text_tool_parsing` defaulting to `True` only for local/non-tool models. For OpenAI-compatible providers, rely on `tool_calls` delta only.

---

**D. `confirm` + `pending_tool_calls` flow is incomplete**

When write tools need confirmation, the agent yields `confirmation_required` and stops. The caller must then call `agent.run(..., confirm=True, pending_tool_calls=...)` but there is no documented API for that. State (history, current iteration count) is not preserved.

**Fix**: Store pending writes as instance state, add an explicit `confirm_pending()` method, or return a continuation object.

---

**E. `max_tokens: int = 2048484` looks like a typo**

In `AgentConfig` you probably meant `2048`. A 2M token limit on a 1B/3B model is nonsensical and will likely crash or be ignored.

---

### 2. `micron/llm.py` — Backends

#### What works
- Clean abstract base class.
- Lazy loading of llama.cpp model with explicit `unload()`.
- Unified `LLMResponse` dataclass.
- Reasoning-content passthrough for DeepSeek/ornith-style models.

#### Issues

**A. llama.cpp chat_format is hard-coded to `chatml`**

```python
Llama(..., chat_format="chatml")
```

Qwen, Llama, Gemma, and SmolLM each need different chat templates. SmolLM2 uses a custom tokenizer template, not ChatML. Using ChatML will degrade output quality and break stop-token handling.

**Fix**: Make `chat_format` a constructor argument and document per-model settings:

```python
MODEL_CHAT_FORMATS = {
    "smollm2": "smollm",
    "qwen2.5": "qwen",
    "llama3.2": "llama-3",
    "gemma2": "gemma",
}
```

---

**B. No request retries or configurable timeouts**

Ollama and OpenRouter HTTP backends have fixed timeouts (300s) and no retry. Local model startup can cause transient failures.

**Fix**: Add `max_retries`, `timeout`, and `retry_delay` to backend constructors.

---

**C. OpenAI backend does not use the official `openai` client**

You manually parse SSE with `requests`. This is error-prone (e.g., `data: [DONE]` handling) and loses built-in retry/backoff.

**Fix**: Use `openai.OpenAI` with `stream=True`, or at least make it an optional dependency. Otherwise `pip install openai` enables it.

---

**D. No structured output / JSON mode support**

For 1B models, forcing JSON output (e.g., `response_format={"type": "json_object"}`) can dramatically improve tool-call reliability. Not currently supported.

---

### 3. `micron/memory.py` — File Memory

#### Issues

**A. IDF precedence bug**

```python
self._idf[term] = math.log(n_docs / df) + 1.0 if df > 0 else 0.0
```

This parses as `(math.log(...) + 1.0) if df > 0 else 0.0`. The `+ 1.0` is inside the truth branch. If you wanted IDF smoothing, it should be:

```python
self._idf[term] = math.log(n_docs / df) + 1.0
```
or
```python
self._idf[term] = math.log((n_docs + 1) / (df + 1)) + 1.0
```

Current behavior is not necessarily wrong, but looks like a typo. Decide whether to use raw IDF (drop `+1`) or smoothed IDF and write a test.

---

**B. Memory has no vector/semantic component**

Keyword search fails for paraphrases. With 1B models you can optionally use lightweight embeddings or stay keyword-only. The current keyword search is fine for the MVP, but consider an optional `sentence-transformers` provider behind a flag.

**Fix** (later): Add `EmbeddingMemory` adapter implementing the same interface.

---

**C. Memory files are written with no locking**

Concurrent CLI/server processes could corrupt JSONL. Use file locking (`portalocker` or `fcntl`) or implement append via atomic rename.

---

### 4. `micron/skills.py` — Skill Loader

#### What works
- Clean YAML frontmatter parser.
- Preserves body content for prompt-based skills.
- `reload()` support.

#### Issues

**A. No JSON schema validation**

A malformed `parameters` block is passed straight to the LLM. This can cause API errors (e.g., missing `type`, invalid `required`).

**Fix**: Validate against a minimal JSON-Schema subset or use `jsonschema` as optional dep.

---

**B. Skill `module` import can crash on file load**

If a skill references a module that exists but whose import has side effects, it can break startup.

**Fix**: Lazy-load module-backed tools on first call, not at agent construction time.

---

### 5. `micron/prompt.py` — Prompt Builder

#### What works
- Dynamic persona loading.
- Knowledge ranking by keyword overlap.
- 8k char budget for knowledge.

#### Issues

**A. Tool instructions are hard-coded and enormous**

The `BASE_TEMPLATE` lists 15 tools by hand, output format examples, boundary rules etc. This is:
1. Fragile — add a tool and the prompt is out of date.
2. Wasteful — many 1B models have 2k–8k context; a 5k-token instruction prompt plus knowledge eats most of it.
3. Wrong — API models don't need the text markup format; they use native function calling.

**Fix**: Make instructions dynamic:

```python
def _load_tools(self):
    lines = []
    for s in self.skills.all():
        marker = " [WRITE-CONFIRM]" if s.write else ""
        params = ", ".join(s.parameters.get("properties", {}).keys())
        lines.append(f"- {s.name}{marker}: {s.description} ({params})")
    return "\n".join(lines)

OUTPUT_INSTRUCTIONS = """If you need to act, call a function. For models that support it, use the function_call protocol. If no tool fits, answer directly."""
```

Keep total system prompt under ~1000 tokens for 1B/3B models.

---

**B. Knowledge fallback is noisy**

If keyword score is 0, you fall back to all knowledge files. For large vaults this can blow the 8k budget with irrelevant content.

**Fix**: If no relevant files, return `(no relevant knowledge)`. Let the model decide whether to use `search_knowledge` or `read_file`.

---

**C. Memory is not included in prompt**

Despite initializing `Memory`, `build_system_prompt` does not inject relevant memories. There is `_load_knowledge` but no memory retrieval.

**Fix**: Add `memory.search(query, k=5)` output block.

### 6. `micron/tools/builtin.py` — Built-in Tools

#### What works
- Broad coverage: web, file, shell, calc, python, memory, knowledge, skill creation.
- Clear string returns (good for LLM consumption).
- Working directory enforced via env var.

#### Issues

**A. Path traversal guard is bypassable**

```python
if not str(target_path).startswith(str(workdir.resolve())):
    return "Error: Security violation."
```

`resolve()` is called on the joined path, so `/tmp/secret` outside workdir is blocked. But symlinks inside workdir can point elsewhere, and `..` chains with unusual separators may slip. Use `os.path.commonpath` or verify `target_path.resolve().is_relative_to(workdir)`.

**Fix**:

```python
from pathlib import Path
workdir = _get_workdir().resolve()
target = (workdir / path).resolve()
try:
    target.relative_to(workdir)
except ValueError:
    return "Error: Security violation."
```

---

**B. `python_eval` is unsafe despite blocklist**

```python
blocked = ["import os", "import subprocess", ...]
exec(code, namespace, namespace)
```

Blocklists are trivially bypassed (`__import__('os')`, `import os as x`, string concatenation). This is sandbox-by-obscurity.

**Fix options**:
- Remove `python_eval` entirely; guide users to create explicitskills.
- Run in a subprocess with restricted seccomp/namespace (Docker).
- Use `asteval` or `RestrictedPython` as a genuine sandbox.
- At minimum mark the tool `write=True` so it requires confirmation.

---

**C. `calculate` uses `eval` with partial namespace**

Better than raw eval, but `__import__` is available inside the namespace via `"math": __import__("math")`. Users can still craft malicious expressions (e.g., via `().__class__.__bases__`).

**Fix**: Use `ast.literal_eval` for simple literals only, or implement a Shunting-yard parser for arithmetic.

---

**D. `run_command` blocklist is incomplete**

```python
blocked = ["rm -rf", "mkfs", "dd if=", ...]
```

Misses `curl | bash`, `wget -O`, `find / -exec rm {} \;`, `python -c`, etc. Also `sudo`, `curl`, `nc`, and many others are dangerous in an agent loop.

**Fix**: Consider an allowlist approach instead: only permit `cat`, `ls`, `pwd`, `grep`, `head`, `tail`, `git status`, `git diff` by default. Add a config flag for unrestricted mode.

---

**E. `save_memory` duplicates logic from `Memory` class**

It reconstructs the memory path and writes JSONL manually. This violates DRY and means memory indexing is not refreshed.

**Fix**: Accept a `Memory` instance or context dir and call `memory.add(...)`.

---

**F. `create_skill` writes YAML manually**

String-building YAML is fragile. Use `yaml.safe_dump` for the frontmatter.

---

**G. Tools return inconsistent types**

Some return `str`, some `dict`, some `list[dict]`, and `read_file` returns formatted strings with `---` headers. This makes it hard for the agent to feed results back into another tool.

**Fix**: Define a `ToolResult` convention:

```python
{"ok": True, "data": ..., "truncated": False, "error": None}
```

Let the LLM-facing layer stringify only when emitting to the model.

---

### 7. `micron/tools/registry.py` — Tool Registry

Good. Minor suggestions:

**A. `auto-detect required` defaults to all params with no default**

This is fine, but does not handle `*args` / `**kwargs`.

**B. Missing `unregister` method**

Useful for hot-reloading skills without restarting.

## Recommended Priority Order

### P0 — Must Fix Before Real Use
1. Correct OpenAI/llama.cpp tool-role message history in `agent.py`.
2. Change default `max_tokens` from `2048484` to `2048`.
3. Replace prompt hard-coded tool list with dynamic generation and make it model-appropriate.
4. Fix IDF calculation precedence or add a comment + test.
5. Harden path traversal using `relative_to` check; consider symlinks.
6. Remove or properly sandbox `python_eval`; don't ship a bypassable blocklist.

### P1 — Strongly Recommended
7. Add a test suite (`pytest`) starting with memory, skill loader, tool registry, and path-traversal checks.
8. Make `chat_format` configurable per model.
9. Add CLI entry point (`micron`) and a way to run one-shot queries.
10. Use official `openai` client for OpenAI-compatible backend.
11. Add tool result envelope (structured response, not raw strings).

### P2 — Nice to Have
12. Optional vector memory with sentence-transformers.
13. Atomic file writes / locking for JSONL memory.
14. Streaming HTTP server with SSE.
15. Config file (`micron.yaml`) instead of env vars.
16. Skill marketplace / templates for common tasks.

## Suggested Refactor: Tool Result Envelope

Move tools toward structured returns:

```python
@dataclass
class ToolResult:
    ok: bool
    data: Any
    display: str       # LLM-readable summary
    error: str | None = None
    truncated: bool = False
```

```python
# Agent side
result = self.tools.call(tc.name, **tc.args)
if isinstance(result, ToolResult):
    content = result.display
else:
    content = json.dumps(result)

messages.append({
    "role": "tool",
    "tool_call_id": tc.call_id,
    "content": content,
})
```

This separates internal data from LLM consumption and makes downstream reasoning more reliable.

## Suggested Refactor: Smaller System Prompt

```python
SYSTEM_HEADER = """You are micron, a lightweight AI assistant. Date: {today}.
You can use functions to act. Only call a function when needed.
If you call functions, the results will be provided to you. Do not hallucinate results."""

TOOL_FOOTER = """Functions:
{tools}
"""
```

For text-based models, append:

```
To call a function output: <function name="tool_name">{"arg": "value"}</function>
```

Keep entire system prompt under ~1500 tokens for 1B/3B models.

## Conclusion

`micron` is a promising foundation — small, understandable, and file-native. The next iteration should focus on **protocol correctness** (tool messages), **security hardening** (path traversal, code execution), and **tests**. Once those are in place, adding a vector memory and config file are natural extensions.

The project is roughly a weekend away from being a solid local/agent-in-a-box tool. Prioritize the P0 items first.
